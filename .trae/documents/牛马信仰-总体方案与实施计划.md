# 《牛马信仰》Windows 桌面应用：总体架构与实施计划（Rust + Tauri）

## 0. 摘要

本项目是一个带“高压监督 + 黑色幽默 + 长期修行体系”的 Windows 桌面自律监督工具。技术上以 **Tauri 2 + Rust（后端）+ Vue3/Pinia（前端）+ SQLite（本地存储）** 为核心，采用 **分层架构（domain / application / data / adapters）+ 事件驱动监督引擎（state machine）**，确保：

- 本地优先、离线可用：任务、计时、监督、扣分、升级/降级/护甲完全本地运行
- 可插拔：AI 与 IM 通知通过 trait/adapter 插拔，外部不可用不影响主流程
- 模块边界清晰：数值系统、规则引擎、监督状态机、数据层分离，便于迭代扩展

本计划既包含你要求的“完整方案输出（架构/模块/模型/状态机/流程/UI/配置/MVP顺序）”，也包含可直接落地到当前仓库的具体改造路径与验证清单。

## 2. 总体架构设计（Rust + Tauri 桌面架构）

### 2.1 分层结构（推荐保持并强化现有结构）

- **domain（纯领域层）**：只放“规则、状态机、数据结构、纯计算”，不做 IO
  - 信仰值计算（1000/天）
  - 等级表（Lv1~Lv15）
  - 扣分规则/护甲规则（纯函数 + 可测试）
  - 监督状态机（状态、事件、转移、产物）
- **application（应用服务层）**：编排领域逻辑 + 调用数据层与 adapter
  - TaskService：任务生命周期与计时、今日汇总
  - SupervisionService：提醒、无输入检测、解释流程、违规生成
  - SettlementService：日结算/周期性结算（工作不足/目标未完成/连续规则）
  - FaithLedgerService：统一“信仰变动记账”（加分/扣分/护甲吸收/降级）
- **data（数据层）**：SQLite schema + repo，实现“事实记录”
  - tasks / task_sessions / daily_records / penalties / faith_transactions / settings ...
- **adapters（外部适配层）**：
  - IdleDetector（Windows GetLastInputInfo）
  - Notifier（Webhook IM）
  - AiClient（OpenAI-compatible / 本地空实现）
- **tauri（接口层）**：
  - commands：前端 invoke API
  - events：后端向前端 emit 监督事件（状态变化、提醒、弹窗触发等）

### 2.2 关键原则：事件驱动监督引擎

监督系统不是“前端定时器堆逻辑”，而应该是一个后端常驻引擎：

- 后端周期 tick（例如 1s/5s）读取：
  - 当前任务状态（running/paused/none）
  - 当前系统 idle 时长（无键鼠输入）
  - 计时与提醒频率
- 状态机决定：
  - 是否发出提醒事件
  - 是否进入“要求解释”状态并触发解释弹窗
  - 是否生成违规记录并触发扣分
- 前端只负责渲染与收集用户解释输入；业务裁决在 domain/application

这样做能保证：

- 规则统一、不因页面切换而丢失
- 离线可运行、可测试
- 后续加入 AI/IM 只要订阅事件即可

## 5. 核心状态机设计（监督系统）

### 5.1 状态定义（最小闭环）

监督状态（对外展示与驱动 UI）建议定义为：

- `NoTask`：当前无进行中任务
- `Focusing`：任务运行中，正常专注
- `Reminder`：到达提醒间隔，发出提醒（不改变任务状态）
- `IdleSuspected`：检测到无输入超过阈值（T1），已提醒“恢复输入”
- `ExplainRequired`：无输入超过更长阈值（T2），必须解释（弹窗）
- `Explaining`：用户正在填写解释
- `Judged`：解释已提交，系统裁决（可能扣分/记中断/恢复任务）
- `Paused`：任务暂停

### 5.2 事件与转移（关键路径）

关键事件：

- `Tick(now, idle_ms)`
- `TaskStarted(task_id)` / `TaskPaused` / `TaskResumed` / `TaskEnded`
- `UserActivityResumed`（idle_ms 回落）
- `ExplainSubmitted(reason, action_after)`（回来后动作）
- `ExplainTimeout`（弹窗超时/拒绝/无反馈，按未解释处理）

关键转移（示例）：

- `Focusing` + `Tick idle_ms >= T1` → `IdleSuspected` 并 emit `reminder`
- `IdleSuspected` + `Tick idle_ms >= T2` → `ExplainRequired` 并 emit `open_explain_modal`
- `ExplainRequired` + `ExplainSubmitted` → `Judged`（生成 violation/penalty）→ 根据 action 返回 `Focusing/Paused/NoTask`
- `ExplainRequired` + `ExplainTimeout` → 生成 `idle_unexplained` violation + 扣分 → `Paused`（默认把任务暂停，避免“空跑”）

### 5.3 输出（后端 emit 给前端）

统一事件 payload（JSON）：

- `supervision/state_changed`：{ state, task_id, idle_ms, message }
- `supervision/reminder`：{ message, severity }
- `supervision/explain_required`：{ task_id, idle_ms, prompt, suggested_reasons[] }
- `ledger/penalized`：{ points, message, today_penalty_total }
- `level/changed`：{ from, to, message, armor_granted }

## 6. 业务流程（任务流 / 监督流 / 扣分流 / 升降级流）

### 6.1 任务流（Work/Study 计时闭环）

1. 创建任务（work/study）
2. 开始任务：
   - 写 tasks.started_at
   - 新增 task_sessions(start_ts)
   - supervision 状态进入 `Focusing`
3. 暂停任务：
   - 结束当前 session（end_ts，秒数入库）
   - tasks.status=paused
   - break_count +1（或由中断定义触发）
4. 恢复任务：
   - 新建 session
   - tasks.status=running
5. 结束任务：
   - 结束 session
   - tasks.ended_at、status=completed
   - 计算任务时长归集到 daily_records.work_minutes/study_minutes
   - 可异步触发 AI 总结（失败不影响结束）

### 6.2 监督流（提醒 → 解释 → 裁决）

1. 后端每 N 秒 Tick：
   - 若无 running task，可降低强度：只做“仪式提醒/提示去开启任务”
   - 若有 running task：
     - idle < T1：正常
     - idle ≥ T1：提醒恢复输入（文案：冷酷系统风）
     - idle ≥ T2：触发解释弹窗（强制解释可配置）
2. 用户提交解释：
   - 写入 violations（kind=break/idle_explain_required + meta）
   - 若理由“发呆/摸鱼”：即时扣分（-200）
   - 若空白/敷衍：即时扣分（-100）
3. 若用户无反馈（超时或忽略）：
   - 写 violations(kind=idle_unexplained)
   - 即时扣分（首次 -300，第二次 -500，第三次起 -800/次）
   - 任务默认暂停（可配置）

### 6.3 扣分流（即时 / 日结算 / 周期性）

统一入口：`FaithLedgerService::apply_penalty(...)`

- 计算扣分点数（按规则表）
- 护甲优先吸收：
  - `armor_used = min(user.armor_points, penalty_abs)`
  - `remaining = penalty_abs - armor_used`
  - `user.armor_points -= armor_used`
  - `user.cumulative_faith -= remaining`
- 写入：
  - penalties（展示明细）
  - faith_transactions（账本）
  - daily_records.penalty_points / net_delta 汇总更新
- 触发降级检查（见 6.4）

### 6.4 升级/降级流（含护甲发放）

升级（加分后可能触发）：

- 每次 `apply_grant`（日信仰/调账）后重新计算 level
- 若 level 提升：
  - 写 `faith_transactions(kind=level_up)`
  - 发放护甲（加入到 armor_points）：
    - Lv2-L5：+2,000
    - Lv6-L10：+4,000
    - Lv11-L15：+6,000
  - emit 升级文案：
    - “你的供奉已被记录。你已由 工位信徒 晋升为 初级供奉者。”

降级（扣分后可能触发）：

- 若 `cumulative_faith < 当前等级门槛`：
  - 重新根据阈值表计算应处 level（Lv1 不可再降）
  - 写 `faith_transactions(kind=level_down)`
  - emit 降级文案：
    - “由于持续失守，你的位阶已受损。你已由 苦修执行官 降为 工时祭司。”

- 学习 0 小时：0
- 学习 2 小时：100
- 学习 4 小时：200
- 学习 6 小时：300
- 学习 8 小时及以上：400

戒律信仰（200）：

- 专注稳定：80
  - 中断 <= 2 次：80
  - 中断 3~4 次：40
  - 中断 >= 5 次：0

  - 无长时间失联：60
  - 有离岗但已反馈并恢复：30
  - 长时间离岗未反馈：0
- 记录闭环：60
  - 完成当日总结 / 状态确认 / 收尾记录：60
  - 未完成：0

1) 即时违规扣分

- A. 长时间离开电脑未反馈
  - 当日首次未反馈离岗：-300
  - 当日第二次未反馈离岗：-500
  - 当日第三次及以上：-800/次
- B. 离岗后反馈但理由恶性
  - 明确摸鱼 / 发呆：-200
  - 敷衍反馈 / 空白说明：-100
- C. 开始任务后短时间放弃
  - 开始后 10 分钟内结束：-150
  - 当日第 3 次后，每次：-250
- D. 高频切换任务
  - 当日切换任务超过 6 次：-200
  - 超过 10 次：额外再 -300

2) 日结算扣分

- A. 工作时长未达标（每日工作目标：8 小时）
  - 工作 6~8 小时：0
  - 工作 4~6 小时：-500
  - 工作 2~4 小时：-1,200
  - 工作 0~2 小时：-2,500
- B. 预期目标未完成（每日 1~3 个核心目标）
  - 完成率 100%：0
  - 完成率 50%~99%：-400
  - 完成率 1%~49%：-1,000
  - 完全未完成：-1,800
- C. 未完成每日闭环：-400
- D. 当天学习完全空白（学习 0 小时）：-300

3) 周期性扣分

- A. 连续 3 天工作不足 6 小时：-1,500
- B. 连续 7 天学习平均低于 1 小时：-1,000
- C. 连续 3 天出现未反馈离岗：-2,000
- D. 连续 7 天核心目标未完成：-3,000

### 7.4 降级与护甲（最终版）

- 降级：当累计信仰低于当前等级门槛，立即降级到对应等级（Lv1 不可再降）
- 晋升护甲：升阶后获得护甲，优先吸收扣分
  - Lv2-L5：护甲 2,000
  - Lv6-L10：护甲 4,000
  - Lv11-L15：护甲 6,000

## 8. 关键页面与交互设计（信息必须体现）

### 8.1 首页（Dashboard / 仪式台）

必须信息：

- 当前任务（标题）
- 当前任务类别（工作/学习）
- 当前状态（专注/提醒/解释中/中断/暂停）
- 今日信仰：X / 1000
- 生存/精进/戒律三项及子项（可折叠）
- 当前位阶：Lv.X + 称号
- 距离下一阶还差多少（progress_to_next）
- 今日扣分与净变动（penalty_points / net_delta）

交互：

- 开始任务 / 暂停 / 恢复 / 结束
- 立即“状态确认/收尾记录”（用于戒律闭环 +60）
- 进入“历史/统计/设置”

文案风格（示例，统一世界观）：

- “检测到持续静默，请恢复输入。”
- “系统需要你解释刚才的离岗行为。”
- “当前产出链路已中断，请重新进入工作状态。”
- “你的沉默已被记录。”
- “你不是消失了，你只是暂时没有创造价值。”

### 8.2 任务页（修行清单）

- 今日任务 + 历史任务筛选
- 创建任务（work/study，标题、备注）
- 任务详情：时长、开始/结束时间、AI总结（可空）
- 支持：开始、暂停、恢复、结束、查看

### 8.3 统计页（圣账统计）

至少包含：

- 今日工作时长、学习时长
- 中断次数、长时间失联次数
- 每日目标完成情况
- 本月累计信仰、本月供奉天数
- 当前等级与总信仰

### 8.4 历史页（审判记录）

- 历史任务
- 历史中断/离岗解释
- 历史扣分记录（可展开查看规则原因）
- 历史升级/降级记录（含护甲发放）

### 8.5 设置页（戒律配置）

- 提醒频率
- 无输入判定阈值（T1/T2）
- 是否开启强制解释
- AI 接口配置（endpoint/key/model）
- IM webhook 配置（url、开关、触发场景）
- 文案风格（冷酷强度：低/中/高，但不辱骂）
- 开机启动（Windows 自启动：后续迭代）

## 10. 可迭代 MVP 范围与开发顺序（保证可运行、可验证、可扩展）

> 说明：你要求“不要省略数值系统”，因此在设计上全量保留；但实现上会按 MVP 分层交付，确保每一步都有可运行闭环。

### MVP1：数值系统切换 + 任务计时闭环（不含监督）

- 将信仰上限切换为 1000/天，等级表切换为 1,095,000
- tasks 支持 start/pause/resume/end，落库 task_sessions
- 首页展示“当前任务/状态/今日信仰/位阶/距离下一阶”
- 引入 faith_transactions 作为统一账本

验收：

- 计时正确累加到 work_minutes/study_minutes
- 每日信仰分项与总计正确（400/400/200）
- 升级/降级计算正确（先不引入扣分）

### MVP2：监督系统（无输入检测 + 解释弹窗 + 即时扣分）

- 后端 IdleDetector（Windows）+ supervision 状态机引擎
- 前端解释弹窗（固定原因 + 自定义）
- 即时违规扣分（A/B/C/D 的即时规则先落地）
- 护甲机制上线（扣分先扣护甲）

验收：

- 无输入达到阈值触发提醒与解释弹窗
- 未反馈按次数扣分；恶性理由/敷衍理由扣分生效
- 扣分后护甲吸收与降级逻辑正确

### MVP3：日结算与周期性扣分 + 历史页

- daily_goals（1~3）与完成率
- 每日闭环（close_record）按钮
- 日结算扣分（工作不足/目标未完成/未闭环/学习空白）
- 周期性扣分（连续规则）在每日结算后评估
- 历史页展示：扣分明细、违规、升降级、护甲消耗

验收：

- 跨天后能对“昨天”完成结算并生成扣分/流水
- 连续规则计算正确（需要测试数据）

### MVP4：IM webhook 单向通知

- WebhookNotifier（失败不阻塞）
- 覆盖场景：开始任务/长时间无输入/离岗未反馈/任务完成/每日总结

### MVP5：AI 接入（可插拔、非阻塞）

- AI 任务拆解（针对任务标题/备注生成 checklist）
- AI 离岗追问文案（更有“审判感”的追问，但不辱骂）
- AI 任务总结（结束任务后异步生成 ai_summary）
- AI 日报/周报生成（基于 daily_records + tasks + violations + penalties）

## 13. 假设与决策（为避免实现阶段反复决策）

- 平台：优先保证 Windows；IdleDetector 使用 Windows API（GetLastInputInfo）；其他平台可用 no-op
- 时间基准：以本地时间（Windows 时区）生成 `YYYY-MM-DD`，不再混用“北京时间”概念
- 单用户：MVP 继续沿用 `default_user`，后续可扩展多用户
- AI：默认 OpenAI-compatible 协议（endpoint/model/key），并提供 Noop 实现；AI 永不阻塞任务/监督主流程
- IM：先做 webhook 单向通知（POST JSON），失败只记录日志不重试阻塞
